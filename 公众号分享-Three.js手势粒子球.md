# 用 Three.js + MediaPipe 做一个可用手势操控的 3D 粒子能量球

> 不需要鼠标，也不需要复杂设备。打开摄像头，张开手掌、捏合手指或握拳，就能控制一个由 3 万颗粒子组成的能量球。

最近用 AI 辅助完成了一个很适合做网页视觉展示的小作品：**手势控制粒子能量球**。

它不是简单地让一个球跟着手移动，而是把手掌位置、手指张合、捏合距离和移动速度映射成不同的粒子参数：移动手掌控制位置，张开手掌让粒子扩散，捏合或握拳压缩能量球，快速挥动则提高旋转速度。

项目只有一个 HTML 文件，核心技术是 Three.js、`THREE.Points`、自定义 Shader，以及 MediaPipe Hands。

## 最终效果包含什么

- 3 万颗独立粒子构成的球体，兼顾外壳和内部体积。
- 冷色、暖色和白色粒子混合，形成能量感。
- 9000 颗远景星点和 1800 颗环形尘埃，增强空间纵深。
- 摄像头实时识别单手 21 个关键点。
- 手掌移动、张开、捏合、握拳和挥动分别控制不同参数。
- 摄像头不可用时，页面仍可进入自动展示模式。

## 可以直接复用的提示词

下面这段提示词适合交给代码模型生成第一版。重点不是只说“做一个粒子球”，而是把粒子数量、交互映射、性能边界和视觉限制写清楚。

```text
请创建一个完整可运行的单文件 HTML，使用 Three.js 和 MediaPipe Hands 实现手势控制的 3D 粒子能量球。

视觉要求：
1. 全屏深黑色宇宙背景，使用固定透视相机。
2. 主体必须由 THREE.Points 构成，包含约 30000 个独立粒子。
3. 粒子同时分布在球体外壳和内部，外壳略密，内部保留体积感。
4. 粒子颜色混合青蓝、白色、橙红和少量金色。
5. 使用 ShaderMaterial 绘制圆形粒子，中心清晰、边缘柔和。
6. 增加远景星空和环形尘埃层，形成空间纵深。

手势要求：
1. 使用 MediaPipe Hands 识别单手 21 个关键点。
2. 手掌中心控制粒子球在画面中的 X、Y 位置。

4. 张开手掌时，粒子向外扩散。
5. 握拳时，粒子球进一步压缩并转为偏暖色。
6. 手掌快速移动时，提高球体旋转速度和能量强度。
7. 所有手势参数必须使用阻尼平滑，避免画面抖动。

技术要求：
1. 使用 BufferGeometry 保存位置、方向、随机种子、尺寸和颜色。
2. 粒子位移在顶点着色器中计算，不要逐帧在 CPU 更新 30000 个粒子。
3. 使用 requestAnimationFrame，限制异常大的帧间隔。
4. 限制 renderer pixelRatio 不超过 2。
5. 摄像头不可用时仍保持自动动画，并显示短暂提示。
6. 输出完整 HTML、CSS 和 JavaScript，不依赖构建工具。
```

## 实现原理一：怎样生成“有厚度”的粒子球

球面上的随机点不能直接随机 X、Y、Z，否则会分布不均。代码先随机球面方向，再决定粒子位于外壳还是内部。

方向生成使用球坐标思路：随机方位角 `theta`，再随机 `z`，由 `sqrt(1-z²)` 得到水平截面的半径。这样得到的方向在球面上更均匀。

```js
const theta = Math.random() * Math.PI * 2;
const z = Math.random() * 2 - 1;
const ring = Math.sqrt(Math.max(0, 1 - z * z));
const dx = Math.cos(theta) * ring;
const dy = Math.sin(theta) * ring;
const dz = z;
```

接下来将 60% 左右的粒子放在半径约 178 的薄壳附近，其余粒子使用 `Math.cbrt(Math.random())` 分布到球体内部。立方根非常重要，它能让体积采样更均匀，避免大量粒子挤在中心。

## 实现原理二：为什么要把动画放进 Shader

如果 JavaScript 每一帧循环更新 3 万个粒子的位置，再把数据上传给 GPU，容易造成明显的 CPU 压力和数据传输开销。

更合适的方案是：初始化时只上传静态属性，动画阶段只修改少量 uniform。

每颗粒子保存以下数据：

- `position`：初始位置。
- `aDirection`：从球心向外的方向。
- `aSeed`：独立随机种子。
- `aSize`：粒子尺寸。
- `aColor`：基础颜色。

顶点着色器根据时间、扩散量、压缩量和能量值计算最终位置：

```glsl
displaced *= mix(1.0, 0.54, uCompression);
displaced += aDirection * wave * (3.0 + uEnergy * 10.0);
displaced += tangent * crossWave * (2.0 + uEnergy * 7.0);
displaced += aDirection * uSpread * (65.0 + aSeed * 240.0);
```

这里同时叠加了径向波动、切线扰动和扩散位移，所以粒子球不会像一个僵硬的点阵模型，而是持续有能量流动。

## 实现原理三：把手势变成稳定参数

MediaPipe Hands 会返回 21 个手部关键点。项目使用腕部和四个指根位置计算手掌中心：

```js
const palm = [landmarks[0], landmarks[5], landmarks[9], landmarks[13], landmarks[17]];
```

为了适配不同手掌大小，捏合和手掌张开程度不能直接使用像素距离，而要除以掌宽做归一化。

### 捏合

拇指尖是关键点 4，食指尖是关键点 8。两点距离除以掌宽后，可以得到相对稳定的捏合比例。

### 张开和握拳

计算食指、中指、无名指和小指指尖到手腕的平均距离，再除以掌宽。数值较大表示张开，较小表示握拳。

### 移动速度

比较当前手掌中心和上一帧的位置差，可以估算挥动速度，并映射到旋转速度。

## 实现原理四：阻尼比识别精度更重要

手势识别结果天然会有小幅抖动。如果直接把识别值赋给球体，画面会不停震动。

代码为每个参数同时保存当前值和目标值，然后使用指数阻尼逐步靠近：

```js
const ease = 1 - Math.exp(-dt * 9.0);
gesture.spread += (gesture.targetSpread - gesture.spread) * ease;
gesture.compression += (gesture.targetCompression - gesture.compression) * ease;
```

这种写法与帧率关系较小，比固定乘以 `0.1` 更稳定。在 60 FPS 和 120 FPS 屏幕上，手感不会差异太大。

## 三种手势对应的视觉反馈

### 张开手掌：扩散

`uSpread` 增大后，粒子沿自身方向外扩，同时叠加切线方向摆动，形成松散的能量爆发感，但不会变成一次性爆炸。

### 捏合或握拳：压缩

`uCompression` 会缩小整体位置和粒子尺寸。压缩时颜色从冷色逐步混合到橙红色，让动作不仅改变形状，也改变能量状态。

### 快速挥动：加速

移动速度映射到 `targetSpin`，球体旋转加快，Shader 中的能量参数也随之提高。

## 背景为什么要分成两层

只有粒子球时，空间感很弱。项目增加了两种背景：

1. 远景星空：粒子位于大半径球形空间中，缓慢旋转。
2. 环形尘埃：分布在主体周围，颜色更鲜艳，旋转方向与星空不同。

两层速度都很慢，作用是提供参照和纵深，而不是抢主体。

## 性能优化清单

- 主粒子、星空和尘埃都使用一次 Draw Call 的 `THREE.Points`。
- 动画位移放在顶点着色器中。
- `devicePixelRatio` 最大限制为 2。
- 粒子材质关闭 `depthWrite`，减少透明粒子的深度冲突。
- 帧间隔限制在 33 毫秒以内，避免切回页面时动画突然跳跃。
- 手势只识别一只手，降低 MediaPipe 计算量。

## 容易踩的坑

### 摄像头权限

摄像头通常要求 HTTPS 或 localhost 环境。直接双击本地 HTML 时，浏览器可能拒绝摄像头权限。开发时建议用本地静态服务器打开。

### 粒子太亮

AdditiveBlending 很容易过曝。颜色、透明度和粒子尺寸要一起控制，不能只靠降低材质 opacity。

### 手势阈值写死

不同人的手掌比例不同。所有距离都应尽量使用掌宽归一化，并给阈值留出过渡区间。

## 结语

这个案例最有价值的部分，不是“做了一个炫酷粒子球”，而是建立了一套可以复用的交互链路：

**摄像头关键点 -> 特征提取 -> 参数归一化 -> 阻尼平滑 -> Shader 视觉反馈。**

换掉粒子球的几何分布，这套结构还可以继续做火焰、星系、花朵、文字、产品轮廓和音乐可视化。
